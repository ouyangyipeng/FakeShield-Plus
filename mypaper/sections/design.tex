\section{Design}
\label{sec:design}

We present four improvements addressing the weaknesses identified in Section~\ref{sec:original}. Each improvement targets a specific limitation while maintaining backward compatibility with the original FakeShield framework.

\subsection{Pipeline Robustness: Automatic Path Mapping and CLIPVisionTower Fix}

The original FakeShield pipeline suffers from two critical robustness issues that prevent seamless cross-environment deployment. We address both issues with lightweight, non-invasive modifications.

\paragraph{Automatic Path Mapping.} The DTE-FDM module outputs absolute file paths from the training environment (e.g., \url{/home/user/FakeShield/dataset/casia1+/0281.jpg}). When the pipeline is deployed in a different environment, these paths become invalid, causing MFLM to fail with file-not-found errors. We introduce an automatic path mapping module (\texttt{fix\_image\_path.py}) that resolves this issue at inference time. Given a DTE-FDM output JSONL file containing paths $P = \{p_1, p_2, \dots, p_n\}$ and a local image directory $D$, the module extracts the filename $f_i = \text{basename}(p_i)$ for each path, recursively searches for $f_i$ in $D$, and replaces $p_i$ with the resolved local path if found. Formally, the resolved path $p'_i$ is:
\begin{equation}
p'_i = \begin{cases}
    \text{find}(f_i, D) & \text{if } \text{find}(f_i, D) \neq \text{None} \\
    p_i & \text{otherwise}
\end{cases}
\end{equation}

The module is integrated into the pipeline via \texttt{run\_pipeline\_improved.sh}, which inserts a path-fixing stage between DTE-FDM and MFLM. This approach eliminates the need for manual \texttt{--image-path} specification, enables seamless cross-environment deployment, and is backward-compatible with existing DTE-FDM outputs.

\paragraph{CLIPVisionTower Loading Fix.} The \texttt{load\_pretrained\_model} function in \texttt{builder.py} contains a model name matching bug that skips vision tower initialization for DTE-FDM. The original condition checks only for ``llava'' in the model name, which fails to match DTE-FDM. We extend the condition to also match ``dte-fdm'' and ``dte\_fdm''. Additionally, we add a safety check that verifies the vision tower is properly loaded by inspecting the \texttt{is\_loaded} flag and the \texttt{vision\_tower} attribute. If either check fails, we explicitly call \texttt{load\_model()} and synchronize the device placement with the main model. This fix ensures that the CLIPVisionTower and its image processor are always properly initialized, regardless of the model name.

\subsection{INT8 Quantization for Memory-Efficient Inference}

The 13B-parameter DTE-FDM requires 6.94 GB of GPU VRAM in FP16 mode, limiting deployment to high-end GPUs. We apply INT8 quantization using the bitsandbytes library~\citep{bitsandbytes} to reduce memory footprint while preserving detection accuracy.

\paragraph{Quantization Strategy.} We apply INT8 quantization through the \texttt{load\_in\_8bit=True} flag in \texttt{load\_pretrained\_model}. This quantizes all linear layers to 8-bit integers while keeping the vision tower and embedding layers in FP16. We also experiment with INT4 quantization using NF4 (Normal Float 4) with double quantization. However, INT4 fails with Out-Of-Memory errors on our 40GB A100 GPU due to intermediate FP32 buffers required during loading, suggesting that INT4 for 13B models requires either 80GB GPUs or CPU offloading.

\paragraph{Device Synchronization and Compatibility.} INT8 quantization with automatic device mapping distributes model layers across available GPUs, causing device mismatch errors when the vision tower and language model reside on different GPUs. We resolve this by forcing single-GPU execution via the \texttt{CUDA\_VISIBLE\_DEVICES} environment variable. We also identify and resolve a version incompatibility between bitsandbytes 0.45.0 and triton 3.6.0 by downgrading to bitsandbytes 0.41.3.post2 and triton 2.1.0, which are compatible with PyTorch 1.13.0 and CUDA 11.7.

\subsection{Extended Domain Tag Generator}

The original DTG classifier supports only three forgery categories (AIGC, DeepFake, Photoshop), which fails to capture emerging manipulation techniques such as Stable Diffusion inpainting, ControlNet-based manipulation, and Midjourney-generated content. We extend the DTG from 3 to 6 classes by expanding the ResNet-50 classification head from 3 to 6 outputs. The new class weights are initialized from the AIGC class (most similar to SD\_inpaint) with small random noise ($\mathcal{N}(0, 0.01)$) for differentiation, and the model is fine-tuned on the SD\_inpaint dataset~\citep{fakeshield2025}. The extended taxonomy includes: AIGC, DeepFake, Photoshop (original classes), and SD\_inpaint, ControlNet, Midjourney (new classes).

\subsection{CLIP Feature-Guided Mask Refinement and IoU Evaluation}

\paragraph{CLIP-Guided Refinement.} We propose a mask refinement module that uses CLIP visual features to compute an anomaly map. The key insight is that tampered regions often exhibit inconsistent CLIP embeddings compared to the rest of the image. We extract CLIP features from intermediate layers (layer -2 for spatial resolution), compute per-patch cosine similarity to the global average feature, and generate an anomaly map $A(x,y) = 1 - \text{cosine\_sim}(f_{x,y}, f_{\text{global}})$. The anomaly map is fused with the original MFLM mask: $M_{\text{refined}} = (1-\alpha) M_{\text{orig}} + \alpha A$, and thresholded at 0.5 to obtain the final binary mask.

\paragraph{IoU Evaluation Framework.} We implement a systematic IoU evaluation framework (\texttt{evaluate\_iou.py}) that loads MFLM predicted masks and ground truth masks, computes IoU and F1 scores for each image, and generates summary statistics (mean, min, max IoU). The framework supports both actual MFLM outputs and synthetic predictions for testing. The IoU metric is defined as:
\begin{equation}
\text{IoU} = \frac{|M_{\text{pred}} \cap M_{\text{gt}}|}{|M_{\text{pred}} \cup M_{\text{gt}}|}
\end{equation}
This framework enables systematic comparison of localization methods and provides quantitative metrics for future improvements.
